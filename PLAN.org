* OpenHands Nix Packaging — Full Project Plan

** Ecosystem Overview

Three GitHub repos, six deployable artifacts:

| Repo                            | What it produces                                           |
|---------------------------------+------------------------------------------------------------|
| =OpenHands/software-agent-sdk=  | 4 PyPI packages (sdk, tools, agent-server, workspace)      |
| =OpenHands/OpenHands-CLI=       | =openhands= CLI binary (TUI)                              |
| =All-Hands-AI/OpenHands=        | V0 server + React frontend + Docker images (legacy)        |

The V1 architecture (SDK + agent-server) is the future. The V0
monolith is deprecated (removal: April 2026). Both currently coexist.

** Proposed Package Tree

#+begin_example
flake.nix
pkgs/
  cli/
    default.nix            # OpenHands CLI (TUI)
  sdk/
    default.nix            # openhands-sdk, tools, agent-server, workspace
  server/

  frontend/

** Phase 1 — SDK + CLI (build first)

*** =pkgs/sdk/default.nix= — Software Agent SDK (4 packages)

Source: =OpenHands/software-agent-sdk= (pin git rev)
Build system: setuptools
Python: >=3.12

Builds four packages with dependency edges:

#+begin_example
openhands-sdk          (standalone: litellm, fastmcp, pydantic, tenacity, ...)
  └─ openhands-tools   (+ bashlex, libtmux, browser-use, tom-swe)
  └─ openhands-agent-server (+ fastapi, docker, sqlalchemy, uvicorn, alembic)
       └─ openhands-workspace (+ pydantic)
#+end_example

*Notable dependencies to handle:*
| Dep              | Status in nixpkgs | Notes                                        |
|------------------+--------------------+----------------------------------------------|
| litellm          | Likely missing     | Heavy; pulls many LLM provider SDKs          |
| fastmcp          | Likely missing     | MCP protocol library, relatively new          |
| browser-use      | Likely missing     | Needs Playwright/Chromium — make optional     |
| tom-swe          | Likely missing     | Theory-of-mind module, new                   |
| agent-client-protocol | Likely missing | ACP spec library                            |
| libtmux          | In nixpkgs         | Should be fine                               |
| fakeredis        | Likely in nixpkgs  | With lua extra                               |

*Package data:* =openhands-agent-server= bundles a Dockerfile,
wallpaper SVG, and VSCode extension JSON files. These need to be
included via =package-data=. The Dockerfile is used at runtime by the
workspace manager to build agent sandbox images.

*Strategy:*
- =openhands-sdk= — =buildPythonPackage= derivation (pure Nix Python library)
- =openhands-tools= — =buildPythonPackage= derivation (pure Nix Python library)
- =openhands-agent-server= — use =uv2nix= with the =uv.lock= file from
  the SDK monorepo; also vend as a Nix Python library
- =openhands-workspace= — =buildPythonPackage= library + =uv2nix= application

Skip =browser-use= initially (only needed for browsing agent, drags
in Playwright). For missing Python packages, package them in Nix.

*** =pkgs/cli/default.nix= — OpenHands CLI

Source: =OpenHands/OpenHands-CLI= (pin git rev)
Build system: hatchling
Python: ==3.12.* (strict pin!)
Entry points: =openhands=, =openhands-acp=

Key deps: =openhands-sdk=, =openhands-tools=, =openhands-workspace=,
=textual=, =typer=, =rich=, =prompt-toolkit=, =httpx=

*Strategy:* Use =uv2nix= with the CLI's =uv.lock=. Wire in our SDK
packages. Straightforward — the hard deps are all in the SDK layer.
Textual and friends are likely already in nixpkgs.

** Phase 2 — V0 Server + Frontend (for Docker images)

*** =pkgs/server/default.nix= — Legacy V0 Server

Source: =All-Hands-AI/OpenHands= (pin git rev)
Build system: poetry-core (with poetry-dynamic-versioning)
Python: >=3.12, <3.14

This is the =openhands-ai= PyPI package — the V0 monolith. It contains:
- =openhands/= — agents (agenthub), controller, server, runtime, LLM, MCP, etc.
- =third_party/= — alternative runtime impls
- =skills/= — markdown microagent files (bundled as package data)

*Massive dependency surface (~80+ deps):*
| Category          | Deps                                                      |
|-------------------+-----------------------------------------------------------|
| LLM providers     | anthropic[vertex], google-cloud-aiplatform, google-genai, boto3, openai |
| Web framework     | fastapi, uvicorn, starlette, sse-starlette, python-socketio |
| Browser           | playwright, browsergym-core                               |
| Infra             | docker, kubernetes, redis, sqlalchemy, asyncpg, pg8000    |
| Data              | numpy, pypdf, pillow, python-docx, python-pptx            |
| Monitoring        | opentelemetry-api, opentelemetry-exporter-otlp-proto-grpc, lmnr |
| Internal          | openhands-sdk, openhands-agent-server, openhands-tools, openhands-aci |

*Key challenges:*
- =openhands-aci= (v0.3.3) is yet another internal package — need to check its deps
- =pythonnet= requires Mono/.NET runtime
- =browsergym-core= + =playwright= needs Chromium
- =poetry-dynamic-versioning= sets version from git tags — need to patch or override
- =google-cloud-aiplatform= is enormous

*Strategy:* Build with =buildPythonPackage= using =poetry-core= backend.
Hardcode version (bypass dynamic versioning). Make heavy optional deps
(vertex AI, playwright, kubernetes) optional where possible. The server
entry point is =uvicorn openhands.server.listen:app=.

*** =pkgs/frontend/default.nix= — React Frontend

Source: =All-Hands-AI/OpenHands= =frontend/= directory
Build system: npm + Vite + React Router
Node: >=22.12

Produces static files in =frontend/build/= that get served by the
Python backend at =/=.

~40 npm dependencies (react, vite, tailwind, monaco-editor, xterm,
socket.io-client, etc.)

*Strategy:* =buildNpmPackage= with the frontend directory as source.
Run =npm run build= which invokes =react-router build=. Output is the
=build/= directory. The =openhands-ui= component library is consumed
via npm (published to npm registry), so we don't need to build it
ourselves.

** Phase 3 — Container Images

*** =images/agent-server.nix= — V1 Agent Server Container

This is the sandbox environment that runs inside Docker for each
agent session. The upstream Dockerfile (in the SDK repo) has two
variants:

- *Full image:* Python + Node.js + Docker-in-Docker + VNC/XFCE desktop
  + OpenVSCode Server + GitHub CLI + Chromium
- *Minimal image:* Python + Node.js + basic tools (headless only)

And two build modes:
- *Source mode:* copies venv with full Python source
- *Binary mode:* PyInstaller-compiled single binary

*What's in the image:*
- Base: =nikolaik/python-nodejs:python3.13-nodejs22= (Debian Bookworm)
- User: =openhands= (UID 10001)
- Tools: git, jq, tmux, build-essential, curl, wget
- Full variant adds: Docker CE, OpenVSCode Server, TigerVNC, XFCE4, noVNC, Chromium, GitHub CLI
- Entry point: =openhands-agent-server= (either via venv or binary)

*Strategy with =dockerTools=:*

Use =dockerTools.buildLayeredImage= to construct the image. This is
the most complex piece:

1. Base layer: Python 3.13, Node.js 22, coreutils, git, tmux, jq, curl
2. Application layer: openhands-agent-server Python package (from pkgs/sdk/)
3. Full variant layers: docker-client, tigervnc, xfce4, novnc, chromium, openvscode-server, gh

The minimal variant skips step 3.

*Challenges:*
- XFCE + VNC + noVNC in a Nix-built image is non-trivial
- OpenVSCode Server is a custom binary download (need =fetchurl=)
- Docker-in-Docker requires the Docker CLI + socket mounting
- The wallpaper SVG and VSCode extensions need to be placed correctly

*Recommendation:* Start with a minimal-source variant. Add full
variant later. The minimal image is sufficient for headless agent
execution (the most common Nix use case).

*** =images/app.nix= — V0 Legacy App Container

This is the user-facing container that runs the web UI + API server.
The upstream Dockerfile:

1. Builds frontend (Node.js stage)
2. Installs Python deps via Poetry (Python stage)
3. Final image: Python 3.13-slim + static frontend + Python venv
4. Entry: =uvicorn openhands.server.listen:app --host 0.0.0.0 --port 3000=

*What's in the image:*
- =python:3.13.7-slim-trixie= base
- User: =openhands= (UID 42420)
- The Python venv with all deps
- Static frontend build at =frontend/build/=
- =skills/= directory
- =pyproject.toml= + =poetry.lock= (read at runtime for version info)
- Environment: =WORKSPACE_BASE=/opt/workspace_base=, =FILE_STORE=local=

*Strategy with =dockerTools=:*

Use =dockerTools.buildLayeredImage=:
1. Base layer: Python 3.13, curl, ssh, sudo
2. App layer: server package (from =pkgs/server/=) + skills/ + frontend build (from =pkgs/frontend/=)
3. Config: user setup, environment variables, entrypoint

*Challenges:*
- The entrypoint.sh does dynamic user creation based on =SANDBOX_USER_ID= — need to replicate or include as script
- Needs =pyproject.toml= and =poetry.lock= in the image (read at runtime)
- Volume mounts for Docker socket and =~/.openhands=

** Dependency Packaging Strategy

Many dependencies won't be in nixpkgs. Suggested approach:

1. *Check nixpkgs first* — use =python312Packages.xyz= where available
2. *Override/add missing ones* — package them in Nix
3. *For heavy deps (litellm, google-cloud-aiplatform):* consider building
   with =buildPythonPackage= from PyPI source, potentially with some
   optional sub-deps disabled
4. *For Playwright/Chromium:* skip initially, add as optional feature flag

** flake.nix Sketch

#+begin_src nix
  {
    description = "Nix packages for OpenHands AI agent platform";

    inputs = {
      nixpkgs.url = "github:NixOS/nixpkgs/nixos-unstable";
      flake-parts.url = "github:hercules-ci/flake-parts";
    };

    outputs = { self, nixpkgs, flake-parts }:
      # use flake-parts
      flake-parts.lib.mkFlake { inherit self; } {
        systems = [ "x86_64-linux" "aarch64-linux" "x86_64-darwin" "aarch64-darwin" ];
        perSystem = { pkgs, system, ... }:
          let
            python = pkgs.python312;

            sdkPackages = pkgs.callPackage ./pkgs/sdk {
              inherit python;
            };

            cli = pkgs.callPackage ./pkgs/cli {
              inherit python sdkPackages;
            };

            # Phase 2
            server = pkgs.callPackage ./pkgs/server {
              inherit python sdkPackages;
            };

            frontend = pkgs.callPackage ./pkgs/frontend { };

            # Phase 3
            images = {
              agent-server = pkgs.callPackage ./images/agent-server.nix {
                inherit sdkPackages;
              };
              app = pkgs.callPackage ./images/app.nix {
                inherit server frontend;
              };
            };
          in {
            packages = {
              default = cli;
              inherit cli;
              inherit (sdkPackages) openhands-sdk openhands-tools
                openhands-agent-server openhands-workspace;
            }
            // (if server != null then { inherit server frontend; } else {})
            // (builtins.mapAttrs (k: v: v) images);
          };
      };
  }
#+end_src

** Implementation Priority

| Phase | Package                | Effort | Notes                                            |
|-------+------------------------+--------+--------------------------------------------------|
|     1 | =pkgs/sdk/=            | Medium | Foundation for everything. Main risk: litellm.   |
|     1 | =pkgs/cli/=            | Low    | Straightforward once SDK builds.                 |
|     2 | =pkgs/server/=         | High   | ~80 deps, pythonnet, poetry-dynamic-versioning.  |
|     2 | =pkgs/frontend/=       | Low    | Standard =buildNpmPackage=.                      |
|     3 | =images/agent-server=  | High   | Complex multi-layer image with VNC/Docker/VSCode.|
|     3 | =images/app=           | Medium | Simpler image, but needs dynamic user creation.  |

Phase 1 gives users the CLI. Phase 2+3 gives fully Nixified Docker
images replacing the upstream Dockerfiles.

** Future Vision

Images should be more Nix-powered — it should be easy to install new
tools to the agent environment in a declarative fashion. Nix naturally
enables this. Longer term, explore a scalable Nix hypervisor approach
for managing agent sandboxes.
